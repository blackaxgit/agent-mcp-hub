# 07 — Post-Filter Findings (false-positive filter + per-finding verification)

RE-REVIEW (round-2, post-hardening). Each of the 17 raw findings from the four teams (Red 5, Green+DevOps 6, Security-CSO 1, Architecture 5) was handed to an independent verification agent that judged whether the defect is `real`, assigned a confidence, and wrote a note. Tagging convention:

- **[CONFIRMED]** — verifier `real=true` with confidence ≥ 70.
- **[NEEDS-HUMAN]** — verifier `real=true` but confidence < 70 (real defect, but severity/reachability is a judgment call).
- **[UNVERIFIED]** — verifier `real=false` (filtered out: not reachable, intentional-and-documented, or capability already inside the trust boundary).

Tally: 7 CONFIRMED, 6 NEEDS-HUMAN, 4 UNVERIFIED (filtered out).

## NEEDS-HUMAN (6)

| Finding | file:line | Sev | Conf | Verifier note (summary) |
|---|---|---|---|---|
| run_all never sets isError on total failure (Arch A1) | src/server.ts:141 | Medium | 68% | Confirmed: run_all (122-142) returns {content} with no isError under any outcome, while the per-agent tool sets isError on non-zero exit (89-106). A client branching on isError cannot distinguish total failure from success. Genuine API-contract asymmetry. Could be read as intentional partial-result aggregation → medium not high. Fix (set isError when no adapter exited 0) correct. |
| No SIGTERM / graceful drain (Green G2) | src/httpServer.ts:43 | Medium | 65% | Confirmed: zero signal handlers in src/ (only per-request server.close()). On SIGTERM, in-flight /mcp requests dropped and detached agent children not signalled. Real but a hardening gap, not a correctness bug: server is stateless so requests self-teardown, and detached children are reaped by cgroup teardown at grace-window end (bounded waste, no true leak). Medium slightly generous but defensible. Fix sound; group-kill tracked child PIDs is the right extra step. |
| Unbounded caller-supplied timeoutMs (Red R5) | src/server.ts:51-56 | Low | 60% | Reachable: timeoutMs flows unclamped into setTimeout (exec.ts:118-121), the only thing freeing a semaphore slot. Valid low-severity hardening gap; clamp is right. Caveats: exploit example is off — Node overflows delays >2^31-1 ms and fires at ~1ms, so MAX_SAFE_INTEGER does NOT hold forever; realistic worst case ~24.8 days. Slot-holding also needs the child to keep running; outsider path needs non-default exposure. |
| Unbounded semaphore waiter memory (Red R3) | src/exec.ts:61-84 | Medium | 58% | Verified: Semaphore.waiters is genuinely uncapped and httpServer enforces no concurrent-request cap or body-size limit. Reachable as a DoS only when the operator binds off the default 127.0.0.1; on localhost the attacker can exhaust the host anyway, so practical severity is below 'medium'. Overstates the semaphore's role — each waiter is a tiny closure; real memory is retained request/prompt buffers. Hardening (max-waiters → 429, request cap, body-size limit) legitimate but should be downgraded. |
| list_agents probe competes for execution semaphore (Arch A3) | src/server.ts:73 | Low | 58% | Verified: list_agents → checkAvailability runs each adapter's --version through the default injected exec (runCommand → withSlot → shared Semaphore, default 4), so probes compete for the same permits as real runs (run_all can saturate all 4 for up to 300s), coupling monitoring latency to execution load. Real but a low-severity design/SLA tradeoff, not a correctness bug — the semaphore is documented as a cap on ALL children, and a --version probe is such a child. Decoupling worth considering; defensible to leave. |
| Unvalidated cwd containment (Red R4) | src/server.ts:50 | Medium→Low | 50% | Mechanism confirmed: cwd is z.string().optional() with zero validation, passed straight to spawn({cwd}); no allowlist/path-resolve anywhere. Impact overstated: (1) HTTP binds 127.0.0.1 by default with MCP_TOKEN auth, so 'remotely reachable on default image' is wrong — reachability needs misconfig; (2) wrapped agents are not sandboxed to cwd, so any authorized caller can already read/write arbitrary paths via the prompt regardless of cwd — a cwd allowlist is weak defense-in-depth, not the actual boundary. Genuine hardening gap; severity closer to low. |
